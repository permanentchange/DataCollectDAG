# 数据采集 DAG 工具需求文档

## 1. 项目目标

开发一个用于数据采集的 DAG 工具。该工具能够根据配置运行指定的数据采集流程，并具备与外部 ROS 系统通讯的能力。

当前阶段，工具需要支持与 ROS1 系统交互；未来应具备迁移到 ROS2 通讯方式的可行性。

本工具的核心定位是数据采集 DAG 工具，不是以 ROS node 为中心设计的业务节点。当启用 ROS1 通讯能力时，工具允许在 ROS graph 中以 ROS node 形式参与通讯。

工具应支持长期运行，允许丢弃部分数据，但不得因为数据处理、推理或写盘阻塞外部数据接收。

---

## 2. 使用场景

工具主要用于自动驾驶、机器人或移动平台的数据采集任务。

典型使用场景包括：

1. 手动启动指定采集流程。
2. 通过外部 ROS 系统启动指定采集流程。
3. 从 ROS1 topic 接入点云、图像、定位、里程计、TF 等数据。
4. 根据配置执行单入口、单出口、无环的数据处理流程。
5. 将采集数据保存为结构化数据。
6. 将采集数据保存为 rosbag。
7. 根据处理节点结果决定是否保存数据。
8. 支持不同业务场景配置不同采集流程。

---

## 3. 系统边界

本工具第一阶段以独立前台程序形式运行。

工具具备与 ROS1 外部系统通讯的能力。ROS 在本工具中的职责包括：

1. 接收外部 ROS 指令。
2. 订阅 ROS topic。
3. 解析 ROS message。
4. 发布状态或结果到 ROS。
5. 提供 ROS service。

ROS 相关逻辑只用于通讯、数据解析和数据发布，不用于其他耗时数据处理流程。

以下能力属于 DAG 中具体处理节点的功能：

1. 时间同步。
2. 运动补偿。
3. 点云处理。
4. 图像处理。
5. 模型推理。
6. 筛选。
7. 结构化数据保存。
8. rosbag 保存。
9. 其他业务处理逻辑。

---

## 4. 启动与控制需求

工具应支持两种启动采集流程的方式：

1. CLI 手动启动。
2. 外部 ROS 系统指令启动。

---

## 4.1 CLI 启动

工具应支持通过命令行参数指定 pipeline 并立即运行。

命令形式示例：

```bash
data_collect_dag --config config.yaml --pipeline data_collect_normal
```

如果未指定结束条件，该 pipeline 可持续运行，直到收到停止信号或发生终止条件。

CLI 启动 pipeline 后，工具仍应能够监听 ROS start / stop / status 指令。

---

## 4.2 ROS 指令启动

工具应支持通过 ROS topic 或 ROS service 接收外部 start 指令。

Start 指令用于启动指定 pipeline。

Start 指令至少应包含：

```text
pipeline 名称
可选运行参数
```

外部系统只指定要启动的采集流程，不维护 session 生命周期。

---

## 4.3 停止方式

工具以前台程序运行时，应支持通过 `Ctrl+C` 停止当前 session。

工具也应支持通过外部 ROS stop 指令停止当前 session。

`Ctrl+C` 和 ROS stop 均属于主动停止。session 成功响应主动停止后，最近 session 状态应为 `STOPPED`。

如果用户使用 `nohup`、systemd、supervisor、Docker 等方式将程序放到后台运行，则 `Ctrl+C` 不再适用，应依赖外部 stop 指令或进程管理工具停止。

收到停止请求后：

1. 当前 session 应停止接收新样本。
2. 未完成样本应丢弃。
3. 已完整写入的数据应保留。
4. 最近 session 状态应记录为 `STOPPED`。

---

## 4.4 Status 指令

Status 指令仅在工具与外部 ROS 系统交互时起作用。

工具应支持通过 ROS topic 或 ROS service 查询或发布状态。

CLI 前台运行时，可通过日志或控制台输出体现运行状态，不强制要求额外 status 指令。

---

## 5. Session 需求

一次采集任务称为一个 session。

每个 session 由工具内部生成唯一 `session_id`。

`session_id` 用于：

1. 输出目录命名。
2. 状态发布。
3. summary 记录。
4. 日志记录。
5. 异常定位。

同一时刻只允许一个 session 运行。

无论当前 session 是由 CLI 启动还是 ROS 指令启动，只要收到新的 start 指令，均按 session 替换处理。

当已有 session 正在运行时收到新的 start 指令：

1. 工具应产生 warning。
2. 当前 session 应立即停止。
3. 当前 session 不再接收新样本。
4. 当前 session 中未完成样本应被丢弃。
5. 已完整写入的数据应保留。
6. 新 session 应立即启动。
7. 旧 session 的结束原因应记录为 `replaced_by_new_start`。
8. 旧 session 的最近 session 状态应记录为 `STOPPED`。

不同 session 的输出数据必须能够区分，不得发生数据归属混淆。

---

## 6. 状态需求

工具状态分为：

```text
工具运行状态
最近 session 状态
```

工具运行状态只包含：

```text
IDLE
RUNNING
```

含义：

```text
IDLE：当前无 session 正在运行。
RUNNING：当前有 session 正在运行。
```

最近 session 状态包含：

```text
STOPPED
COMPLETED
FAILED
```

含义：

```text
STOPPED：最近 session 响应主动停止或被新 start 替换后成功结束。
COMPLETED：最近 session 因满足结束条件而正常结束。
FAILED：最近 session 异常结束。
```

当 session 结束后，工具运行状态应回到 `IDLE`，同时保留最近 session 状态。

当工具与 ROS 系统交互时，应能够发布或查询状态。

状态信息至少应包含：

```text
工具运行状态
最近 session 状态
当前 session_id
当前 pipeline 名称
开始时间
最近错误
warning 数量
已接收数据数量
已处理样本数量
已保存样本数量
已丢弃样本数量
丢弃原因统计
```

---

## 7. 输入数据需求

工具第一阶段应支持从 ROS1 系统接入以下数据。

### 7.1 点云

```text
sensor_msgs/PointCloud2
```

### 7.2 图像

```text
sensor_msgs/Image
sensor_msgs/CompressedImage
```

### 7.3 定位

```text
geometry_msgs/PoseStamped
nav_msgs/Odometry
```

第一阶段不要求支持自定义 localization message。

### 7.4 里程计

```text
nav_msgs/Odometry
```

### 7.5 TF

```text
/tf
/tf_static
```

### 7.6 标定与固定参数

第一阶段不要求支持 `sensor_msgs/CameraInfo`。

相机内参、畸变参数、外参、固定坐标变换等参数可由具体处理节点按需从本地固定参数文件读取。

---

## 8. 数据接入与缓存需求

工具应对外部输入数据进行有限缓存。

默认缓存要求：

```text
点云：5 帧
图像：5 帧
定位：100 帧
里程计：100 帧
```

`/tf` 与 `/tf_static` 应支持独立缓存或查询能力。

缓存满时应丢弃最旧数据。

外部数据接收路径只应完成：

```text
接收数据
解析 ROS message
提取必要 metadata
写入缓存
快速返回
```

外部数据接收路径不得执行：

```text
模型推理
点云处理
图像压缩
运动补偿
复杂同步逻辑
大文件写盘
```

允许因缓存满、处理能力不足、筛选规则、session 停止或 session 替换而丢弃数据或样本。

不允许在未记录原因的情况下静默丢弃已进入保存流程的有效样本。

---

## 9. DAG 流程需求

工具应支持通过配置文件定义不同的数据采集 DAG 流程。

一个采集流程应由以下元素组成：

```text
开始条件
输入数据
处理节点
节点依赖关系
结束条件
输出方式
运行参数
```

采集流程应为单入口、单出口、无环处理图。

处理图应满足：

```text
只允许一个起点
只允许一个终点
允许分支
允许汇合
允许异步执行
不允许环
不允许多个入口
不允许多个最终出口
```

结构化保存节点、rosbag 保存节点、metadata 保存节点、调试保存节点等可以同时存在，但它们不等同于 DAG 的最终终点。DAG 最终仍应汇合到唯一终点。

工具应支持由 pipeline 配置定义主帧来源。第一阶段至少应支持图像主帧。

---

## 10. 节点依赖需求

处理流程中的每个节点应具有唯一 `node_id`。

`node_id` 用于：

```text
配置引用
依赖关系表达
日志记录
错误定位
状态追踪
```

节点依赖关系应通过显式 edges 表达。

edges 只表达执行依赖，不表达具体数据内容。

节点是否可以执行由 edges 决定，不由 context key 是否已有数据决定。

如果某个节点依赖多个前置节点，则该节点必须等待所有前置节点完成后才能执行。

---

## 11. 数据传递需求

处理节点之间应通过上下文传递数据。

数据传递应满足：

1. 节点输入通过配置声明。
2. 节点输出通过配置声明。
3. 输入输出使用 context key 表达。
4. 不应依赖隐式未声明数据。
5. 应尽量传递引用或轻量对象，避免无意义复制点云、图像等大对象。
6. 默认允许多个节点写入同一个 context key。
7. 当多个节点写入同一个 context key 时，下游节点应能够区分不同 producer 的输出。
8. 节点是否就绪由 edges 决定，不由 context key 写入情况决定。

---

## 12. 处理节点能力需求

工具应允许通过配置选择不同类型的处理节点。

处理节点可用于完成以下类别的功能：

```text
数据解析
时间同步
运动补偿
点云处理
图像处理
模型推理
数据筛选
数据保存
状态统计
结果发布
```

每类功能的详细行为由具体处理节点定义。

需求文档只要求工具具备承载这些处理节点的能力，不规定每个处理节点的内部实现方式。

---

## 13. 时间同步类节点需求

工具应允许在 DAG 中配置时间同步类节点。

时间同步类节点可用于：

```text
以图像为主帧匹配点云
以点云为主帧匹配图像
匹配定位信息
匹配里程计信息
匹配 TF 信息
筛选时间差满足要求的数据
```

具体同步策略由对应节点配置定义。

---

## 14. 运动补偿类节点需求

工具应允许在 DAG 中配置运动补偿类节点。

运动补偿类节点可用于将点云或其他传感器数据补偿到目标时间戳。

具体补偿方式由对应节点配置定义。

---

## 15. 点云处理类节点需求

工具应允许在 DAG 中配置点云处理类节点。

点云处理类节点可覆盖以下能力：

```text
PointCloud2 解析
ROI 裁剪
距离筛选
坐标变换
运动补偿
多帧拼接
格式转换
点云模型推理
```

具体处理逻辑由对应节点配置定义。

---

## 16. 图像处理类节点需求

工具应允许在 DAG 中配置图像处理类节点。

图像处理类节点可覆盖以下能力：

```text
Image 解析
CompressedImage 解析
JPEG 压缩
resize
crop
去畸变
图像质量筛选
图像模型推理
检测结果筛选
```

具体处理逻辑由对应节点配置定义。

---

## 17. 推理类节点需求

工具应允许在 DAG 中配置推理类节点。

推理类节点应支持以下类型：

```text
PyTorch 推理
TensorRT 推理
```

推理类节点可用于：

```text
点云模型推理
图像模型推理
推理结果后处理
基于推理结果控制后续保存或筛选
```

具体模型加载、推理输入、推理输出和后处理方式由对应节点配置定义。

---

## 18. 筛选类节点需求

工具应允许在 DAG 中配置筛选类节点。

筛选类节点可根据以下信息产生筛选结果：

```text
图像质量
点云点数
点云距离范围
检测类别
检测分数
检测数量
保存频率
正样本保存策略
负样本抽样保存策略
其他自定义条件
```

筛选结果应能被后续节点使用。

第一阶段不要求实现复杂规则引擎。

---

## 19. 保存类节点需求

工具应允许在 DAG 中配置保存类节点。

保存类节点可包括：

```text
结构化数据保存
rosbag 保存
metadata 保存
调试数据保存
```

用户可通过 DAG 配置选择：

```text
只保存结构化数据
只保存 rosbag
同时保存结构化数据和 rosbag
```

保存类节点应能够根据上游筛选结果决定是否保存当前样本。

保存类节点应支持配置 required inputs 与 optional inputs。

当 required inputs 缺失时，保存类节点不应生成有效样本。

---

## 20. 结构化数据保存需求

结构化数据格式为：

```text
bin + jpg + json
```

结构化数据保存节点应支持按样本组织数据。

一个样本内可包含一个或多个传感器数据文件及对应 metadata。

结构化数据应能保存：

```text
点云
图像
时间戳
定位信息
里程计信息
TF 或外参相关信息
推理结果
筛选结果
样本 metadata
```

结构化样本 metadata 应支持记录：

```text
传感器名称
topic 名称
时间戳
frame_id
样本主时间
数据文件路径
定位或里程计信息
坐标系相关信息
筛选结果
推理结果
```

如果同时保存 rosbag 和结构化数据，结构化样本 metadata 应能追溯到对应 rosbag、topic 和 timestamp。

---

## 21. rosbag 保存需求

rosbag 保存应支持以下模式：

```text
保存原始 topic
保存处理后 topic
同时保存原始 topic 和处理后 topic
```

具体保存内容由保存类节点配置决定。

---

## 22. 数据完整性需求

工具应避免产生半成品数据。

当 session 被 stop、替换或异常终止时：

```text
已完整写入的数据应保留
未完成样本应丢弃
未完成文件不应作为有效数据出现
```

不同 session 的输出数据应相互隔离。

已完成样本、未完成样本、被丢弃样本应在数据组织和统计记录中可区分。

---

## 23. 配置文件需求

工具只使用一个 YAML 配置文件。

配置文件应至少包含：

```text
基础参数
ROS 通讯配置
缓存配置
模型配置
pipeline 配置
输出配置
状态发布配置
```

Pipeline 配置应能表达：

```text
source
nodes
edges
sink
end_condition
```

工具启动或加载配置时，应对配置文件进行校验。

配置校验至少包括：

```text
pipeline 名称存在
node_id 唯一
edges 引用的节点存在
处理图无环
处理图只有一个入口和一个终点
节点 type 可识别
节点 inputs / outputs 格式合法
```

第一阶段要求启动时完整校验配置文件中的所有 pipeline；任一 pipeline 配置错误，工具启动失败。

---

## 24. 结束条件需求

工具应支持以下 session 结束条件：

```text
外部 stop 指令
Ctrl+C
运行时长达到配置值
样本数量达到配置值
采集流程主动结束
发生致命错误
被新的 start 指令替换
```

结束状态规则：

```text
外部 stop 指令结束：STOPPED
Ctrl+C 结束：STOPPED
被新的 start 指令替换：STOPPED
运行时长达到配置值：COMPLETED
样本数量达到配置值：COMPLETED
采集流程主动正常结束：COMPLETED
发生致命错误：FAILED
```

---

## 25. Summary 需求

每个 session 结束后应生成 `session_summary.json`。

`session_summary.json` 至少包含：

```text
session_id
pipeline 名称
开始时间
结束时间
结束状态
结束原因
配置文件路径
输出目录
```

还应包含：

```text
接收数据数量
进入 DAG 的样本数量
处理样本数量
保存样本数量
失败样本数量
跳过样本数量
丢弃样本数量
丢弃原因统计
warning 数量
error 数量
```

如包含推理或筛选流程，应支持记录：

```text
筛选前样本数量
筛选后样本数量
各类筛选原因统计
推理命中数量
主要类别命中统计
```

---

## 26. 异常处理需求

处理失败应由对应节点或流程定义处理策略。

工具应支持以下失败处理结果：

```text
忽略错误并继续
跳过当前样本
当前样本失败
当前 session 失败
取消当前 session
```

基本要求：

```text
单帧失败不应导致工具退出
模型加载失败可导致当前 session 失败
磁盘写入失败可导致当前 session 失败
session 被替换时未完成样本应取消
长期运行时异常不应导致工具无控制退出
```

---

## 27. 长期运行需求

工具应满足长期运行要求：

```text
外部数据接收不被处理流程阻塞
缓存容量有上限
处理失败可控
状态可查询或发布
session 结束后资源可释放
模型资源可复用
数据保存不产生半成品
允许丢帧
不要求严格实时性
```

工具应具备基本可观测性，至少包括：

```text
运行日志
状态查询或发布
session_summary.json
错误原因记录
丢弃原因统计
每个 session 的配置记录
```

---

## 28. 技术约束

工具主体使用 Python 实现。

当前外部通讯环境：

```text
Ubuntu 20.04
ROS1
Python
```

未来目标外部通讯环境：

```text
Ubuntu 22.04
ROS2
Python
```

工具应允许在性能不足时接入 C++ 实现的处理节点。

---

## 29. 非目标

第一阶段不要求支持：

```text
多个 session 并发运行
多个 source
多个 sink
有环处理图
运行时修改处理图
分布式调度
跨机器任务编排
复杂规则引擎
完整任务管理平台
外部 task 生命周期管理
sensor_msgs/CameraInfo 接入
自定义 localization message 接入
```

---

## 30. 最小可用版本需求

最小可用版本应满足：

```text
独立前台程序运行
支持 CLI 指定 pipeline 启动
支持 CLI 启动后监听 ROS 指令
支持 ROS1 通讯
支持 ROS start / stop / status 交互
支持 Ctrl+C 停止
支持一个 YAML 配置文件
支持配置启动校验
支持 session_id 自动生成
支持 session 替换
支持工具运行状态与最近 session 状态
```
